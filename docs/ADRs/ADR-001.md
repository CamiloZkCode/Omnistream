# ADR 001: Estrategia de Migración del Motor de Descuentos

## Estado
Aceptado

## Contexto

OmniStream es actualmente un sistema monolito que ha estado en producción por 6 meses. La gerencia ha firmado un contrato que multiplicará el tráfico por 100 durante el próximo mes. El análisis de la Fase 1 identificó los siguientes problemas en el MotorDescuentos:

1. **Alto acoplamiento:** El MotorDescuentos depende directamente del objeto `User` mediante `getattr()`, lo que crea una dependencia oculta.
2. **Números mágicos:** Los valores 20, 10 y 25 están escritos directamente en el código sin constante que los explique.
3. **Escalabilidad limitada:** Al estar dentro del monolito, el MotorDescuentos no puede escalar independientemente del resto del sistema.

El negocio requiere:
- El sistema no puede detenerse durante la migración (zero downtime)
- El MotorDescuentos debe poder escalar horizontalmente para soportar el aumento de tráfico
- Las pruebas TDD existentes (G6) deben seguir pasando en verde

## Decisión

**Utilizaremos el Patrón Strangler Fig (Higo Estrangulador) para migrar progresivamente el MotorDescuentos a un microservicio independiente.**

La migración se realizará en tres fases:

1. **Fase 1 (Interceptación):** El monolito redirige las llamadas al MotorDescuentos hacia un proxy que aún ejecuta la lógica local.
2. **Fase 2 (Migración):** Se implementa el microservicio externo y el proxy desvía el tráfico gradualmente (20% → 50% → 100%).
3. **Fase 3 (Estrangulación):** Se elimina el código del MotorDescuentos del monolito original.

## Opciones Consideradas

### Opción A: Reescritura Total (Big Bang)
Detener el sistema, reescribir el MotorDescuentos como microservicio, y desplegar todo de nuevo.

### Opción B: Strangler Fig (seleccionada)
Migración incremental manteniendo el sistema funcionando.

## Argumentos

### Ventajas del Strangler Fig
| Aspecto | Beneficio |
|---------|------------|
| **Riesgo** | Bajo. El monolito sigue funcionando durante toda la migración |
| **Rollback** | Inmediato. Basta con cambiar el proxy para que use la implementación local |
| **Validación** | Las pruebas TDD existentes verifican que el contrato no se rompe |
| **Aprendizaje** | El equipo aprende gradualmente la operación del microservicio |

### Desventajas del Strangler Fig
| Aspecto | Inconveniente |
|---------|---------------|
| **Complejidad temporal** | Se mantienen dos implementaciones (local y remota) durante la transición |
| **Overhead operativo** | Se requiere un proxy o feature flag para el enrutamiento |
| **Tiempo total** | Mayor tiempo hasta la eliminación completa del código legado |

### Comparación con Big Bang

| Criterio | Strangler Fig | Big Bang |
|----------|---------------|----------|
| Tiempo de implementación | 3-5 días | 1-2 días |
| Riesgo de downtime | Cero (migración en caliente) | Alto (requiere ventana de mantenimiento) |
| Capacidad de rollback | Inmediato (cambiar flag) | Complejo (redeploy completo) |
| Validación continua | Sí (tests pasan todo el tiempo) | No (todo o nada) |
| Esfuerzo total | Mayor (2 implementaciones temporales) | Menor (solo la nueva) |

## Trade-offs Identificados

1. **Riesgo vs. Tiempo:** Big Bang es más rápido pero más riesgoso. Strangler Fig es más seguro pero requiere más esfuerzo total. Dado que el sistema no puede detenerse, priorizamos la seguridad sobre la velocidad.

2. **Complejidad vs. Escalabilidad:** Mantener ambas implementaciones durante la transición aumenta la complejidad temporal, pero permite validar el microservicio con tráfico real antes de comprometerse completamente.

## Consecuencias

### Positivas
- Zero downtime durante la migración
- Las pruebas TDD existentes validan continuamente la corrección
- Se puede hacer canary testing (desviar solo un porcentaje del tráfico al microservicio)
- El equipo adquiere experiencia operativa con microservicios de forma controlada

### Negativas
- Se requiere implementar un mecanismo de enrutamiento (proxy o feature flag)
- Durante la Fase 2, hay código duplicado (lógica local + cliente HTTP)
- El despliegue inicial del microservicio requiere configuración adicional de red

### Neutrales
- Se necesitarán monitoreo y logging diferenciados para comparar respuestas entre ambas implementaciones

## Relación con Code Smells identificados en Fase 1

Este ADR mitiga específicamente:
- **Code Smell #2 (Estado global mutable):** El microservicio tendrá su propia configuración aislada
- **Code Smell #5 (Lógica duplicada):** Se centralizará la selección de implementación en un proxy
- **Code Smell #6 (Presentación conoce infraestructura):** El microservicio expone API limpia, el monolito no necesita conocer su implementación interna
